## 2.1 에이전트, 서브에이전트, 오케스트레이션 개념

### [개념] 헷갈리기 쉬운 3가지 — 비유로 이해하기

이 3가지는 성격이 다릅니다. 혼동하기 쉬우니 비유부터 잡고 볼게요.

| 개념 | 비유 | 실제 역할/기능 |
|------|------|--------------|
| 에이전트 | 독립적인 역할과 업무를 가진 정규 팀원 | 독립적인 역할과 프롬프트를 가진 1급 실행 단위. 각자의 전문 분야에서 스스로 판단하며 작업합니다 |
| 서브에이전트 | 팀원이 바쁠 때 잠깐 도움 받는 임시 지원 인력 | 메인 대화의 컨텍스트를 보호하기 위해 임시로 생성되는 작업자. 작업이 끝나면 사라집니다 |
| 오케스트레이션 | 팀장이 정규 팀원과 임시 인력을 배치하고 조율하는 것 | 여러 에이전트/서브에이전트의 작업 순서, 역할 분담, 결과 종합을 관리 |

위 개념들이 실제로 어떻게 동작하는지, 1.4에서 체험한 카드뉴스 자동화 파이프라인을 예시로 살펴볼게요.

이 시스템에서 에이전트, 서브에이전트, 오케스트레이션이 각각 어떤 역할을 하는지 하나씩 살펴볼게요.

---

### 1) 에이전트 (Agent) — 독립적인 1급 실행 단위

위 표에서 Max, Aria, Luna가 각각 에이전트입니다. "카드뉴스 만들어줘"라고 요청하면, Aria가 스스로 데이터를 읽고, 정렬하고, 디자인하고, 이미지를 생성합니다. 사람이 매 단계를 지시하지 않아도 다음 할 일을 스스로 판단합니다.

핵심: 에이전트는 독립적인 역할과 프롬프트를 가집니다.

- 자기만의 전문 영역이 있습니다 (Max는 기획, Aria는 콘텐츠, Luna는 검수)
- 스스로 판단해서 작업을 수행합니다
- 에이전트 = AI + 도구(Tools) + 판단력(Reasoning)

### 2) 서브에이전트 (Sub-Agent) — 임시 작업자

위 표에서 Aria의 세부 내용에 "필요시 서브에이전트로 분리"라고 적혀 있습니다. 서브에이전트는 에이전트가 작업 중 필요에 따라 임시로 생성하는 작업자입니다.

핵심: 서브에이전트는 임시입니다. 메인 대화의 컨텍스트를 보호하기 위해 만들어집니다.

- 메인 대화가 너무 길어지는 것을 방지합니다
- 무거운 작업을 분리해서 처리합니다
- 작업이 끝나면 결과만 남기고 사라집니다

카드뉴스 제작 과정에서 데이터 수집, 이미지 생성, 캡션 작성 등 여러 작업을 동시에 처리해야 합니다. 이때 Aria가 모든 것을 혼자 하면 대화가 길어지고 맥락이 흐려질 수 있습니다. 그래서 Aria는 필요에 따라 서브에이전트를 임시로 생성해서 무거운 작업을 분리합니다. 예를 들어 데이터 분석은 서브에이전트에게 맡기고, Aria는 결과만 받아서 이미지 생성에 집중하는 방식입니다. 서브에이전트는 작업이 끝나면 사라지고, Aria의 메인 대화는 깔끔하게 유지됩니다.

### 3) 오케스트레이션 (Orchestration) — 전체 조율

위 표 전체가 오케스트레이션입니다. Max → Aria → Luna → 퍼블리시 순서로 누가, 언제, 무엇을 할지 전체 흐름을 설계하고 조율하는 것입니다.

#### 오케스트레이션 패턴 5가지

에이전트들을 어떻게 조합하느냐에 따라 다양한 패턴이 있습니다.

| 패턴 | 설명 | 예시 |
|------|------|------|
| Fan Out<br/>(분산 처리) | 같은 작업을 여러 파일에 동시 적용 | "15개 파일에서 각각 핵심 내용을 뽑아서 하나로 합쳐줘" |
| Specialized Roles<br/>(역할 분담) | 다른 역할의 에이전트가 각자 전문 분석 | "에이전트 1은 인터뷰 분석, 에이전트 2는 설문 분석, 에이전트 3은 리뷰 분석" |
| Parallel Research<br/>(병렬 조사) | 여러 대상을 동시에 조사 | "경쟁사 5곳을 동시에 리서치해서 비교표 만들어줘" |
| Batch Generation<br/>(대량 생성) | 비슷한 문서를 대량 생성 | "이 PRD를 기반으로 사용자 스토리 20개 만들어줘" |
| Validation Pipeline<br/>(검증 파이프라인) | 여러 문서를 품질 기준으로 검수 | "docs 폴더의 모든 문서를 품질 체크리스트 기준으로 검토해줘" |

#### 종합(Synthesis)이 핵심!

에이전트 작업이 끝나면 반드시 결과를 종합하는 단계가 필요합니다.

> 에이전트들이 각자 작업 완료
> → "모든 결과를 종합해서 summary.md로 정리해줘"
> → 개별 결과에서 보이지 않던 패턴이 종합에서 드러납니다

> 날것의 에이전트 결과물은 재료일 뿐 — 종합(Synthesis)에서 진짜 가치가 나옵니다.

> 그렇다면 이 에이전트들에게 "무엇을 해야 하는지"를 어떻게 지시할까요? 그것이 바로 PRD(제품 요구 사항 문서)입니다. PRD에 대해서는 2.2에서 자세히 다룰게요.
