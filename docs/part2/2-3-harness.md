## 2.3 하네스 엔지니어링

### [개념] 하네스 엔지니어링이란?

하네스(Harness)란 AI 모델이 혼자서는 잘 못하는 부분을 보완하기 위한 운영 구조입니다. 같은 AI 모델이라도, 어떤 구조 안에서 일하느냐에 따라 결과가 완전히 달라집니다. 이 운영 구조를 설계하는 것을 하네스 엔지니어링이라고 합니다.

그렇다면 왜 이런 구조가 필요할까요? AI에게 일을 그냥 시키면 어떤 문제가 생기는지부터 살펴볼게요.

---

### [개념] AI에게 일을 그냥 시키면 생기는 문제들

> 아래 내용은 Anthropic이 자체 기술 블로그에서 직접 공개한 내용입니다. (2026년 3월 24일 발행)
> 원문: https://www.anthropic.com/engineering/harness-design-long-running-apps

문제 1: 컨텍스트 불안 — 대화가 길어지면 AI가 대충 마무리한다

AI에게 긴 작업을 시키면, 처음에는 꼼꼼하게 수행합니다. 하지만 대화가 길어질수록 앞에서 무엇을 했는지 점점 헷갈리기 시작하고, 실제로 한계에 도달하지 않았더라도 AI가 "이쯤이면 한계에 가까워졌겠지"라고 느끼면서 서둘러 끝내버리는 현상이 발생합니다. Anthropic은 이 현상을 "컨텍스트 불안(Context Anxiety)"이라고 부릅니다.

문제 2: 자기 평가의 함정 — AI에게 자기 결과물을 평가하라고 하면 항상 후하게 준다

AI에게 콘텐츠를 만든 후 "방금 만든 결과물을 평가해봐"라고 요청하면, 품질이 부족하더라도 "아주 훌륭합니다!"라고 답하는 경우가 많습니다. 자기가 만든 것을 객관적으로 비판하는 것은 AI에게도 어려운 일입니다. 특히 디자인처럼 주관적인 영역에서 이 문제가 더 심하게 나타납니다.

---

### [개념] 해결책: 교대 근무 + Generator/Evaluator 역할 분리

해결책 1. 컨텍스트 불안 → Context Reset (교대 근무)

AI 에이전트 하나가 처음부터 끝까지 다 하지 않고, 구간을 나눠서 여러 에이전트가 교대로 작업합니다.

에이전트가 일정 구간을 작업한 뒤, 진행 상태와 다음 할 일을 파일로 저장하고 컨텍스트를 완전히 초기화합니다. 그러면 새로운 에이전트가 해당 파일을 읽고 깨끗한 상태에서 이어서 작업합니다.

교대한 에이전트는 이전 컨텍스트가 없는 깨끗한 상태에서 시작하기 때문에, 뒷부분을 대충 처리하는 문제가 사라집니다.

> 참고: Context Reset 외에 Compaction(압축)이라는 방식도 있습니다. Context Reset은 에이전트를 교체해서 깨끗한 상태로 시작하는 것이고, Compaction은 같은 에이전트가 계속 일하면서 이전 대화를 요약·압축하는 것입니다. Compaction은 대화의 연속성은 유지되지만, 컨텍스트 불안 자체를 해결하지는 못합니다.

해결책 2. 자기 평가 편향 → Generator/Evaluator 분리

설계하는 AI(Generator)와 채점 및 검수하는 AI(Evaluator)를 완전히 분리합니다.

여기서 한 가지 의문이 생길 수 있습니다. "채점하는 AI도 어차피 AI인데, 역시 후하게 주지 않을까요?"

맞는 지적입니다. 하지만 Anthropic이 발견한 핵심 인사이트는 이것입니다:

> "자기 작업을 스스로 비판하도록 만드는 것"보다
> "별도의 채점자를 엄격한 기준으로 세팅하는 것"이 훨씬 쉽습니다.

콘텐츠를 만든 AI에게 "객관적으로 평가해봐"라고 하면 어렵지만, 콘텐츠 제작에 관여하지 않은 별도의 AI에게 명확한 채점 기준을 주고 "이 기준에 맞는지 검수해줘"라고 하면 훨씬 신뢰할 수 있는 결과가 나옵니다.

핵심은 Evaluator에게 구체적인 평가 기준을 정해주는 것입니다. "잘 만들었는지 봐줘"가 아니라, "금지어가 포함되어 있는지, 면책 문구가 있는지, 이미지 사이즈가 1080x1080인지 확인해줘"처럼 명확한 기준을 제시해야 합니다.

### [개념] 실전 패턴: 3-에이전트 구조 (Planner / Generator / Evaluator)

위에서 다룬 문제와 해결책을 하나의 구조로 만든 것이 3-에이전트 패턴입니다. AI에게 복잡한 작업을 시킬 때, 하나의 AI에게 모든 것을 맡기는 대신 기획/실행/검수 3단계로 역할을 나누면 컨텍스트 불안과 자기 평가 편향을 동시에 해결할 수 있습니다.

Planner(기획자) → Generator(실행자) → Evaluator(검수자)
*(기준 미달 시 Generator에게 재작업 요청)*

| 역할 | 하는 일 |
|------|--------|
| Planner (기획자) | 사용자의 요청을 구체적인 작업 명세로 변환합니다 |
| Generator (실행자) | 실제 콘텐츠를 생성합니다 |
| Evaluator (검수자) | 결과물을 평가하고, 기준에 미달하면 재작업을 요청합니다 |

> 이전 섹션(2.1)에서 살펴본 엘라의 마케팅 에이전트 시스템에서는: Max = Planner, Aria = Generator, Luna = Evaluator 역할을 맡고 있습니다.

### [개념] 핵심 교훈 3가지

| 교훈 | 내용 |
|------|------|
| 1. 에이전트를 직접 관찰하라 | AI가 작업하는 과정을 직접 지켜보세요. 어디서 막히는지, 어떤 판단을 내리는지 확인하고, 원하는 결과가 나올 때까지 CLAUDE.md나 프롬프트를 조정하세요. |
| 2. 복잡한 작업은 분해하고 전문화하라 | 하나의 AI에게 모든 것을 시키는 대신, 기획/실행/검수처럼 역할별로 전문 에이전트를 배치하면 전체 성능이 올라갑니다. |
| 3. 새 모델이 나오면 하네스 구조(Planner/Generator/Evaluator)를 다시 점검하라 | AI 모델이 발전하면 불필요해진 구성요소는 제거하고, 새롭게 가능해진 기능을 위한 구성요소를 추가하세요. 점검하지 않으면 불필요한 절차만 쌓이게 됩니다. |
