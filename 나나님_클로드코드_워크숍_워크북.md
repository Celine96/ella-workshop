## 1.1 클로드 코드란?

### [개념] 클로드 코드 = 내 컴퓨터에서 직접 일하는 AI 비서

일반적인 AI 챗봇(ChatGPT 등)은 **대화만** 해요.
클로드 코드는 달라요. 내 컴퓨터의 파일을 직접 읽고, 만들고, 수정하고, 명령을 실행해요.

알아야 할 명령어는 딱 하나: `claude`

### [개념] ChatGPT와 뭐가 다른가요?

| 기능 | ChatGPT                                  | 클로드 코드 | 비고 |
|------|------------------------------------------|-----------|------|
| 파일 접근 (File Access) | 불가                                     | 내 컴퓨터 파일을 직접 읽고 수정할 수 있음 | |
| 기억력 (Memory) | 대화가 끝나면 사라짐                       | 프로젝트 설정 파일에 저장해서 다음에도 기억함 | 2부 2.2 |
| 실행력 (Execution) | 지시한 작업을 실행                         | 지시한 목표를 달성 | |
| 동시 작업 (Parallel) | 한 번에 하나씩 처리                        | 여러 작업을 동시에 처리 가능 | |
| 에이전트 (Agent) | 단일 인스턴스                              | 역할별 전문 에이전트로 분업 가능 | 2부 2.1 |

각 기능을 좀 더 풀어볼게요:

1. 파일 접근 (File Access): ChatGPT는 대화창 안에서만 동작하지만, 클로드 코드는 내 컴퓨터의 폴더와 파일을 직접 다룰 수 있어요.

2. 기억력 (Memory): ChatGPT는 대화를 닫으면 이전 맥락을 잊지만, 클로드 코드는 CLAUDE.md라는 설정 파일에 프로젝트 정보를 저장해서 다음 대화에서도 기억해요.<br>
*(2부 2.2에서 자세히 다룰 예정입니다.)*

3. 실행력 (Execution): ChatGPT는 이메일 전송, 알람 생성처럼 지시한 단일 작업을 실행하지만, 클로드 코드는 지시한 목표를 스스로 분해하고 기획부터 구현까지 자동화해요.<br>
예를 들어 "마케팅 카드뉴스를 만들어줘"라고 하면, 데이터 분석 → 디자인 → 이미지 생성까지 스스로 판단하며 워크플로우를 구축합니다.

4. 동시 작업 (Parallel): ChatGPT는 하나의 요청을 순서대로 처리하지만, 클로드 코드는 여러 작업을 동시에 처리할 수 있어요.

5. 에이전트 (Agent): ChatGPT는 하나의 AI가 모든 것을 처리하지만, 클로드 코드는 기획·실행·검수처럼 역할별 전문 에이전트를 두고 분업할 수 있어요.<br>
*(2부 2.1에서 에이전트 개념으로 자세히 다룰 예정입니다.)*

> 핵심 차이: ChatGPT도 실무에 활용할 수 있지만, 새 대화를 시작할 때마다 프로젝트 맥락, 선호사항, 용어 등을 매번 다시 설명해야 합니다. 이것을 재설명 문제(Re-Explanation Problem)라고 해요. 클로드 코드는 CLAUDE.md에 이 정보를 저장해두기 때문에, 매번 처음부터 설명할 필요 없이 바로 작업을 이어갈 수 있습니다.

### [개념] 마인드셋 전환

|  | 기존 방식 | AI-Native 방식 |
|:---:|:---|:---|
| 사고방식 | "프로그래밍 언어를 배우고, 라이브러리를 익혀서 직접 구현해야 해" / "GPT 프롬프트를 잘 써야 좋은 결과가 나와" | "목적을 명확하게 전달하고, 역할을 분담시키면 AI가 구현해줘" |
| 구현 방식 | 사람이 코드를 한 줄씩 작성 / 프롬프트를 반복 수정하며 결과물 추출 | AI 에이전트가 기획부터 구현까지 자동으로 수행 |
| 학습 내용 | Python, JavaScript 등 코딩 언어 학습 / GPT 프롬프트 작성법 공부 | 원하는 결과를 정확하게 설명하는 방법 / AI에게 역할을 나눠주는 방법 |

이 워크숍에서 배울 것은 코딩이 아니에요. 목적을 명확하게 정의하고, AI에게 역할을 분담시켜서 기획부터 구현까지 자동화하는 방법을 배워요.

## 1.2 설치 및 첫 실행

### [실습] 클로드 코드 설치하기

Step 1. 터미널을 열어주세요.
- Windows: 시작 메뉴에서 "Windows PowerShell"을 검색하여 실행해 주세요. (명령 프롬프트(cmd)가 아닌 PowerShell을 선택해 주세요.)
- Mac: Spotlight(Cmd+Space)에서 "터미널"을 검색하여 실행해 주세요.

Step 2. 아래 설치 명령어를 복사해서 터미널에 붙여넣고 Enter를 눌러주세요.

Windows PowerShell:
```
irm https://claude.ai/install.ps1 | iex
```

Mac/Linux:
```
curl -fsSL https://claude.ai/install.sh | bash
```

Mac (Homebrew 사용자):
```
brew install --cask claude-code
```

Step 3. 설치가 완료되면 터미널을 닫았다가 다시 열어주세요. (이 단계가 중요해요!)

### [실습] 첫 실행 & 로그인

Step 1. `claude --version`을 입력해서 설치가 정상적으로 되었는지 확인해 주세요.

```
claude --version
```

> 버전 번호가 표시되면 설치 완료예요. "command not found"가 나오면 터미널을 다시 닫고 열어주세요.

Step 2. 작업할 폴더를 만들고 이동해 주세요.

Windows PowerShell:
```
mkdir C:\Users\나나\Desktop\워크숍
cd C:\Users\나나\Desktop\워크숍
```
> "나나" 부분은 본인의 Windows 사용자명으로 바꿔주세요. 사용자명을 모르겠다면 파일 탐색기에서 `C:\Users` 폴더를 열어 본인 폴더명을 확인할 수 있어요.

Mac/Linux 터미널:
```
mkdir ~/Desktop/워크숍
cd ~/Desktop/워크숍
```

Step 3. 클로드 코드를 실행해 주세요:

```
claude
```

Step 4. 로그인해 주세요.
- "Claude account with subscription"을 선택해 주세요. (Anthropic Console이 아닌, 구독 계정 옵션을 선택하세요.)
- 브라우저가 열리면 Claude 계정으로 로그인해 주세요.
- 로그인 완료 후 터미널로 돌아와 주세요.
- 텍스트 스타일을 선택하라는 안내가 나오면, 원하는 스타일을 선택해 주세요.

Step 5. 아래 화면이 나오면 성공이에요:

```
╭──────────────────────────────────────╮
│ Welcome to Claude Code!              │
│                                      │
│ /help for available commands         │
╰──────────────────────────────────────╯

You:
```

## 1.3 명령어 & 네비게이션

### [개념] 슬래시 명령어 한눈에 보기

클로드 코드 안에서 `/`로 시작하는 명령어를 사용할 수 있어요. 전부 외울 필요는 없고, 필요할 때 이 표를 참고하세요.

**세션 관리 (가장 많이 씀)**

| 명령어 | 하는 일 |
|--------|--------|
| `/clear` | 대화를 초기화합니다. 파일은 그대로 유지됩니다 |
| `/compact` | 대화가 길어졌을 때 압축합니다. 토큰을 절약하고 클로드가 앞부분을 까먹는 걸 방지합니다 |
| `/resume` | 클로드 코드를 껐다 켜도 이전 대화를 복원합니다. 퇴근 후 다음날 이어서 작업할 때 유용합니다 |

**모델 & 사용량**

| 명령어 | 하는 일 |
|--------|--------|
| `/model` | 모델을 전환합니다. Sonnet(빠름) ↔ Opus(똑똑함) |
| `/cost` | 현재 세션의 비용을 확인합니다 |
| `/usage` | 전체 사용량을 확인합니다 |

**컨텍스트 & 파일**

| 명령어 | 하는 일 |
|--------|--------|
| `/context` | 클로드가 현재 인식하고 있는 파일 목록을 봅니다 |
| `/add-dir` | 참고할 폴더를 추가합니다 |
| `/init` | 새 프로젝트에서 CLAUDE.md를 자동 생성합니다 |
| `/memory` | 프로젝트 기억(메모리)을 확인하거나 수정합니다 |

**에이전트 & 권한**

| 명령어 | 하는 일 |
|--------|--------|
| `/agents` | 서브에이전트를 생성하고 관리합니다 |
| `/permissions` | 도구 사용 권한을 설정합니다 |

**기타**

| 명령어 | 하는 일 |
|--------|--------|
| `/help` | 사용 가능한 모든 명령어를 봅니다 |
| `/config` | 설정을 변경합니다 |
| `/doctor` | 문제가 있을 때 진단합니다 |
| `/quit` 또는 `/exit` | 클로드 코드를 종료합니다 |

### [개념] 3가지 입력 모드

`Shift+Tab`을 누르면 모드가 전환돼요.

| 모드 | 설명 | 언제 쓰나요? |
|------|------|-------------|
| **Default (기본)** | 클로드가 작업하기 전에 매번 허락을 구합니다. 읽기 작업만 자동 실행됩니다 | 처음 배울 때 (기본값) |
| **Accept Edits (자동 승인)** | 파일 편집과 일반적인 파일 작업을 자동으로 승인합니다 | 익숙해진 후, 신뢰할 수 있는 작업 |
| **Plan (계획)** | 클로드가 먼저 계획을 보여주고, 승인하면 실행합니다 | 복잡한 작업을 단계별로 확인하고 싶을 때 |

> 처음에는 **Default 모드**(기본값)로 사용하면서 하나씩 확인하는 걸 권장해요.

### [개념] 키보드 단축키

| 단축키 | 하는 일 |
|--------|--------|
| `Escape` | 클로드가 응답 중일 때 중단시킵니다 |
| `Escape` × 2번 | 되감기 — 대화만 되돌리거나, 파일 변경까지 되돌릴 수 있습니다 |
| `Shift+Tab` | 입력 모드를 전환합니다 (Default → Accept Edits → Plan) |

| `Ctrl+V` | 이미지를 붙여넣습니다 |

## 1.4 실전: 부동산 실거래가 카드뉴스 만들어보기

> 엘라(이 파이프라인을 만든 기획자)가 만든 **부동산 실거래가 카드뉴스 파이프라인**의 간소화 버전을 직접 체험해 봐요.

### [개념] 파이프라인 흐름

```
CSV 데이터          필터 & 선별           HTML 카드 디자인        결과물
┌──────────┐     ┌──────────────┐     ┌───────────────┐     ┌──────────┐
│ 서울 전체  │ ──→ │ 26년 실거래가  │ ──→ │ 1080x1080     │ ──→ │ 카드뉴스  │
│ 실거래가   │     │ 최고가 + 최저가│     │ 인스타 카드    │     │ 이미지    │
│ 데이터     │     │ 각 1건 선별   │     │ PNG 생성     │     │ 완성!    │
└──────────┘     └──────────────┘     └───────────────┘     └──────────┘
```

엘라의 실제 시스템에서는 이 과정이 전부 자동으로 돌아가요.
오늘은 클로드 코드에게 직접 요청해서 이 흐름을 체험해 볼게요.

### [준비] 실습 파일 세팅

아래 파일들을 워크숍 폴더에 미리 넣어주세요. (사전에 공유된 자료예요.)

```
📁 워크숍 폴더/
├── 실거래가_단계적매칭_서울전체_완전판_오차10%.csv   ← 실거래가 데이터
└── templates/                                      ← 카드뉴스 HTML 템플릿
    ├── card_cover.html
    └── card_data.html
```

### [개념] 사용할 데이터

CSV 파일: `실거래가_단계적매칭_서울전체_완전판_오차10%.csv`

이 파일에는 서울 전체 상업용 부동산 실거래 데이터가 들어 있어요.

**데이터에 담긴 정보:**
- 시군구, 지번 주소, 도로명 주소, 건축물 용도, 거래금액, 전용/연면적, 대지면적, 건축년도 등

**이 데이터가 만들어진 과정:**

국토교통부에서 제공하는 실거래가 데이터에는 한 가지 문제가 있어요. 개인정보 보호를 위해 지번이 마스킹(***) 처리되어 있어서, 정확히 어떤 건물이 거래된 것인지 알 수 없어요.

> 예시: 강남구 청담동 *** 번지 → 어떤 건물인지 특정 불가

이 문제를 해결하기 위해, 건축물대장 표제부라는 공적 장부를 활용했어요. 표제부에는 건물의 정확한 주소와 상세 정보가 기록되어 있는데요, 실거래가 데이터와 표제부 데이터에서 공통으로 존재하는 3가지 정보(연면적, 대지면적, 건축년도)를 기준으로 대조하면 마스킹된 지번과 도로명 주소를 복원할 수 있어요.

다만, 실제 데이터에서는 면적이 정확히 일치하지 않는 경우도 있어요. 예를 들어 건물이 코너에 위치하면 실제 사용 가능한 대지면적이 공부상 면적과 다를 수 있고, 옥탑 등 일부 공간이 면적에 포함되지 않는 경우도 있어요. 그래서 면적에 약간의 오차 범위(±10%)를 허용하는 방식으로 매칭을 진행했어요.

오늘 활용할 데이터는 위의 전처리가 완료된 일부 데이터예요.

이 데이터에서 **2026년 실거래가 중 최고가와 최저가** 각 1건을 뽑아 카드뉴스 샘플을 만들어 볼게요.

오늘 사용할 2건은 다음과 같아요:

| 구분 | 주소 | 거래금액 |
|------|------|---------|
| **최고가** | 서울특별시 마포구 양화로 141 (동교동 160-5번지) | **2,650억원** |
| **최저가** | 서울특별시 구로구 도림로10길 23 (구로동 791-58번지) | **1억 6,500만원** |

> 같은 "서울 상업용 부동산"인데 금액 차이가 엄청나죠?
> 이렇게 극단적인 두 건을 카드로 만들면, 데이터의 힘을 한눈에 보실 수 있어요.

### [실습] 건물 사진 준비하기

카드뉴스에 들어갈 건물 사진을 미리 준비해 주세요.

**Step 1.** 아래 2건의 건물 사진을 인터넷에서 찾아 캡처해 주세요:

| 구분 | 주소 | 검색 키워드 |
|------|------|-----------|

| 최저가 | 구로구 도림로10길 23 | "구로구 도림로10길 23" 또는 로드뷰에서 캡처 |

**Step 2.** 워크숍 폴더 안에 `images` 폴더를 만들어 주세요.

# 2부: 에이전트 구축 & 자동화 완성하기

## 2.1 에이전트, 서브에이전트, 오케스트레이션 개념

### [개념] 헷갈리기 쉬운 3가지 — 비유로 이해하기

이 3가지는 **성격이 다릅니다.** 혼동하기 쉬우니 비유부터 잡고 볼게요.

| 개념 | 비유 | 실제 역할/기능 |
|------|------|--------------|
| **에이전트** | 독립적인 역할과 업무를 가진 정규 팀원 | **독립적인 역할과 프롬프트를 가진 1급 실행 단위.** 각자의 전문 분야에서 스스로 판단하며 작업합니다 |
| **서브에이전트** | 팀원이 바쁠 때 잠깐 도움 받는 임시 지원 인력 | 메인 대화의 **컨텍스트를 보호하기 위해 임시로 생성**되는 작업자. 작업이 끝나면 사라집니다 |
| **오케스트레이션** | 팀장이 정규 팀원과 임시 인력을 배치하고 조율하는 것 | 여러 에이전트/서브에이전트의 **작업 순서, 역할 분담, 결과 종합**을 관리 |

---

### 하나씩 자세히 볼게요.

### 1) 에이전트 (Agent) — 독립적인 1급 실행 단위

```
"카드뉴스 만들어줘"라고 말하면
→ 클로드가 스스로: 데이터 읽기 → 정렬 → 디자인 → 이미지 생성
→ 사람이 매 단계를 지시하지 않아도, 다음 할 일을 스스로 판단합니다
→ 이것이 "에이전트"입니다
```

**핵심:** 에이전트는 **독립적인 역할과 프롬프트**를 가집니다.

- 자기만의 **전문 영역**이 있습니다 (예: Max는 기획, Aria는 콘텐츠, Luna는 검수)
- 스스로 판단해서 작업을 수행합니다
- **에이전트 = AI + 도구(Tools) + 판단력(Reasoning)**

### 2) 서브에이전트 (Sub-Agent) — 임시 작업자

```
클로드가 카드뉴스를 만들다가, 복잡한 데이터 분석이 필요해지면:
  "이 부분은 별도로 처리하자" → 임시 작업자를 생성해서 분석만 맡김
→ 분석이 끝나면 결과만 가져오고 임시 작업자는 사라집니다
→ 메인 대화는 깔끔하게 유지됩니다
```

**핵심:** 서브에이전트는 **임시**입니다. 메인 대화의 컨텍스트를 보호하기 위해 만들어집니다.

- 메인 대화가 너무 길어지는 것을 **방지**합니다
- 무거운 작업을 **분리**해서 처리합니다
- 작업이 끝나면 결과만 남기고 **사라집니다**

### 3) 오케스트레이션 (Orchestration) — 전체 조율

```
[오케스트레이터]
    ├── Max 에이전트: "오늘 할 일 계획해"          → 독립 에이전트가 기획
    ├── Aria 에이전트: "카드뉴스 만들어"            → 독립 에이전트가 제작
    │       └── (Aria가 필요시 서브에이전트 생성)   → 임시 작업자가 데이터 분석
    ├── Luna 에이전트: "품질 검수해"                → 독립 에이전트가 검수
    └── 최종 게시
```

**누가, 언제, 무엇을 할지** 전체 흐름을 설계하고 조율하는 것입니다.

---

### [개념] 엘라의 마케팅 에이전트에서 이 구조 찾기

```
오케스트레이션 (main.py — 전체 흐름 관리)
    │
    ├── Max (매니저 에이전트) ─── 오늘 할 일 계획
    │
    ├── Aria (콘텐츠 에이전트) ─── 카드뉴스 제작
    │       ├── 데이터 수집 (필요시 서브에이전트로 분리)
    │       ├── 이미지 생성 (필요시 서브에이전트로 분리)
    │       └── 캡션 작성
    │
    ├── Luna (검수 에이전트) ─── 품질 체크
    │       ├── 금지어 확인
    │       └── 법률 준수 확인
    │
    └── 퍼블리시 ─── 인스타그램 게시
```

Max, Aria, Luna는 각각 **독립적인 에이전트**입니다. 작업 중 무거운 처리가 필요하면 **서브에이전트를 임시로 생성**해서 분리 처리하고, 전체 순서는 main.py가 **오케스트레이션**합니다.

---

### [개념] 오케스트레이션 패턴 5가지

에이전트들을 어떻게 조합하느냐에 따라 다양한 패턴이 있습니다.

| 패턴 | 설명 | 예시 |
|------|------|------|
| **Fan Out (분산 처리)** | 같은 작업을 여러 파일에 동시 적용 | "15개 파일에서 각각 핵심 내용을 뽑아서 하나로 합쳐줘" |
| **Specialized Roles (역할 분담)** | 다른 역할의 에이전트가 각자 전문 분석 | "에이전트 1은 인터뷰 분석, 에이전트 2는 설문 분석, 에이전트 3은 리뷰 분석" |
| **Parallel Research (병렬 조사)** | 여러 대상을 동시에 조사 | "경쟁사 5곳을 동시에 리서치해서 비교표 만들어줘" |
| **Batch Generation (대량 생성)** | 비슷한 문서를 대량 생성 | "이 PRD를 기반으로 사용자 스토리 20개 만들어줘" |
| **Validation Pipeline (검증 파이프라인)** | 여러 문서를 품질 기준으로 검수 | "docs 폴더의 모든 문서를 품질 체크리스트 기준으로 검토해줘" |

### 종합(Synthesis)이 핵심!

에이전트 작업이 끝나면 반드시 **결과를 종합하는 단계**가 필요합니다.

```
에이전트들이 각자 작업 완료
    ↓
"모든 결과를 종합해서 summary.md로 정리해줘"
    ↓
개별 결과에서 보이지 않던 패턴이 종합에서 드러납니다
```

> 날것의 에이전트 결과물은 재료일 뿐 — **종합(Synthesis)에서 진짜 가치가 나옵니다.**

> 그렇다면 이 에이전트들에게 "무엇을 해야 하는지"를 어떻게 지시할까요? 그것이 바로 **PRD(제품 요구 사항 문서)**입니다. PRD에 대해서는 2.3에서 자세히 다룰게요.

### [체크포인트]
- [ ] 에이전트(독립 실행 단위) / 서브에이전트(임시 작업자) / 오케스트레이션(조율)의 차이를 이해함
- [ ] 엘라의 시스템에서 각 개념이 어디에 해당하는지 파악함

## 2.2 하네스 엔지니어링

### [개념] 하네스 엔지니어링이란?

하네스(Harness)란 AI 모델을 감싸는 **운영 구조**입니다. 같은 AI 모델이라도, 어떤 구조 안에서 일하느냐에 따라 결과가 완전히 달라집니다. 이 운영 구조를 설계하는 것을 **하네스 엔지니어링**이라고 합니다.

그렇다면 왜 이런 구조가 필요할까요? AI에게 일을 그냥 시키면 어떤 문제가 생기는지부터 살펴볼게요.

---

### [개념] AI에게 일을 그냥 시키면 생기는 문제들

> 아래 내용은 Anthropic이 자체 기술 블로그에서 직접 공개한 내용입니다.
> 원문: https://www.anthropic.com/engineering/harness-design-long-running-apps

**문제 1: 컨텍스트 불안 — 대화가 길어지면 AI가 대충 마무리한다**

AI에게 긴 작업을 시키면, 처음에는 꼼꼼하게 수행합니다. 하지만 대화가 길어질수록 앞에서 무엇을 했는지 점점 헷갈리기 시작하고, 아직 할 일이 남았는데도 "이쯤이면 됐겠지"하며 서둘러 끝내버리는 현상이 발생합니다. Anthropic은 이 현상을 **"컨텍스트 불안(Context Anxiety)"**이라고 부릅니다.

**문제 2: 자기 평가의 함정 — AI에게 자기 결과물을 평가하라고 하면 항상 후하게 준다**

AI에게 콘텐츠를 만든 후 "방금 만든 결과물을 평가해봐"라고 요청하면, 품질이 부족하더라도 "아주 훌륭합니다!"라고 답하는 경우가 많습니다. 자기가 만든 것을 객관적으로 비판하는 것은 AI에게도 어려운 일입니다. 특히 디자인처럼 주관적인 영역에서 이 문제가 더 심하게 나타납니다.

---

### [개념] 해결책: 교대 근무 + 별도 검수자

**컨텍스트 불안의 해결 — Context Reset (교대 근무)**

하나의 AI가 모든 작업을 처음부터 끝까지 하는 대신, 작업을 구간별로 나누어 여러 AI가 교대로 수행합니다.

```
AI 1번: 첫 번째 구간 작성
    → "여기까지 완료했고, 다음에는 이 부분을 이어서 해야 합니다" 인수인계 메모 작성
        → AI 2번: 인수인계 메모를 읽고 다음 구간 작성
            → AI 3번: 마지막 구간 마무리
```

각 AI는 항상 머리가 맑은 상태에서 시작하기 때문에, 뒷부분을 대충 처리하는 문제가 해소됩니다.

> 참고로 Compaction(압축)이라는 다른 방식도 있습니다. 이 방식은 같은 AI가 계속 일하면서 앞부분을 요약해 줄이는 것입니다. 대화의 연속성은 유지되지만, AI가 느끼는 "머리가 꽉 차가는 느낌"은 해소되지 않습니다.
> **Compaction = 같은 방에서 짐을 정리하는 것 / Context Reset = 깨끗한 새 방으로 이사하는 것**

**자기 평가 실패의 해결 — Generator/Evaluator 분리**

만드는 AI(Generator)와 채점하는 AI(Evaluator)를 완전히 분리합니다.

여기서 한 가지 의문이 생길 수 있습니다. "채점하는 AI도 어차피 AI인데, 역시 후하게 주지 않을까요?"

맞는 지적입니다. 하지만 Anthropic이 발견한 핵심 인사이트는 이것입니다:

> "자기 작업을 스스로 비판하도록 만드는 것"보다
> **"별도의 채점자를 엄격한 기준으로 세팅하는 것"**이 훨씬 쉽습니다.

콘텐츠를 만든 AI에게 "객관적으로 평가해봐"라고 하면 어렵지만, 별도의 AI에게 명확한 채점 기준을 주고 "이 기준에 맞는지 검수해줘"라고 하면 훨씬 신뢰할 수 있는 결과가 나옵니다.

### [개념] 하네스의 3가지 핵심 요소

이러한 문제들을 해결하기 위해 하네스는 다음 3가지 요소로 구성됩니다.

```
┌──────────────────────────────────────────────────────┐
│                  하네스 (Harness)                      │
│                                                       │
│  (1) CLAUDE.md      = 업무 매뉴얼                     │
│      "이 프로젝트에서 넌 이런 역할이야"                 │
│      "이런 규칙을 따라야 해"                           │
│      "이런 도구를 사용할 수 있어"                      │
│                                                       │
│  (2) System Prompt   = 기본 성격 설정                  │
│      "넌 부동산 마케팅 전문가야"                       │
│      "항상 한국어로 응답해"                            │
│      (CLAUDE.md가 프로젝트별 매뉴얼이라면,             │
│       System Prompt는 AI의 기본 성격을 정하는 것입니다. │
│       이 워크숍에서는 CLAUDE.md가 두 역할을 겸합니다.)  │
│                                                       │
│  (3) Tools & Hooks   = 도구와 자동 검수 장치           │
│      "파일 읽기/쓰기 가능"                            │
│      "이미지 생성 가능"                               │
│      "작업 완료 시 자동으로 품질 체크 실행"             │
│      (Hooks란, 특정 시점에 자동으로 실행되는 규칙입니다. │
│       예: 파일 저장 시 자동으로 금지어 검사를 실행)      │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### [개념] 실전 패턴: 3-에이전트 구조 (Planner / Generator / Evaluator)

위에서 다룬 문제와 해결책을 하나의 구조로 만든 것이 3-에이전트 패턴입니다.

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   Planner    │ ──→ │  Generator   │ ──→ │  Evaluator   │
│   (기획자)   │     │  (실행자)     │     │  (검수자)     │
│              │     │              │ ←── │              │
│ "무엇을 할지 │     │ "실제로      │     │ "품질 확인 후 │
│  계획 수립"  │     │  만든다"     │     │  통과/반려"   │
└──────────────┘     └──────────────┘     └──────────────┘
```

- **Planner(기획자)** — 사용자의 요청을 구체적인 작업 명세로 변환합니다.
- **Generator(실행자)** — 실제 콘텐츠를 생성합니다.
- **Evaluator(검수자)** — 결과물을 평가하고, 기준에 미달하면 Generator에게 재작업을 요청합니다.

> 다음 섹션(2.3)에서 자세히 살펴볼 엘라의 마케팅 에이전트 시스템에서는: **Max = Planner, Aria = Generator, Luna = Evaluator** 역할을 맡고 있어요.

## 2.3 PRD & 에이전트 정의서

### [개념] PRD(Product Requirements Document)란?

2.1에서 에이전트의 개념을 배웠고, 2.2에서 하네스를 통해 AI의 성능을 끌어올리는 방법을 배웠어요. 그런데 한 가지 빠진 것이 있어요. **에이전트에게 "무엇을 해야 하는지"를 어떻게 정확하게 전달할 것인가?** 라는 문제예요.

AI에게 "알아서 카드뉴스 만들어줘"라고 하면 어떻게 될까요? 어떤 데이터를 쓸지, 어떤 디자인으로 만들지, 어떤 기준으로 품질을 판단할지 모두 AI가 임의로 결정하게 돼요. 결과물이 매번 달라지고, 원하는 품질을 안정적으로 얻기 어려워요.

이 문제를 해결하는 것이 바로 **PRD(Product Requirements Document)**입니다. PRD는 AI 에이전트에게 주는 **상세 업무 지시서**로, 다음과 같은 핵심 항목들을 담고 있습니다:

| 항목 | 역할 | 예시 |
|------|------|------|
| **목적** | 이 시스템이 왜 필요한지 | "마케팅 콘텐츠를 자동 생성하여 인력 비용을 절감한다" |
| **목표 지표** | 성공을 어떻게 측정할지 | "월간 신규 관심 유저 수" |
| **에이전트 구성** | 누가 어떤 역할을 맡는지 | "기획자, 실행자, 검수자로 분리" |
| **파이프라인** | 작업이 어떤 순서로 흐르는지 | "데이터 수집 → 선별 → 디자인 → 검수 → 게시" |
| **입출력 정의** | 무엇을 넣고 무엇이 나오는지 | "CSV 데이터 → 1080x1080 카드뉴스 PNG" |
| **품질 기준** | 어떤 기준을 지켜야 하는지 | "금지어 미포함, 면책 문구 필수, 이미지 사양 준수" |

PRD가 잘 작성되어 있으면, AI 에이전트는 매번 일관된 품질의 결과물을 만들어낼 수 있어요.

---

### [개념] 엘라의 PRD 살펴보기 — 1부에서 체험한 작업의 뒷면

1부(1.4)에서 직접 체험한 **'부동산 실거래가 카드뉴스 디자인 파이프라인'** 기억하시나요? 실거래가 CSV 데이터를 넣으면 카드뉴스 PNG가 나오는 그 작업이에요.

그 작업이 안정적으로 돌아가려면 뒤에서 PRD가 모든 것을 정의하고 있어야 해요. 엘라의 PRD 핵심 구조를 살펴볼게요.

```
1. 개요
   └─ 목적: 부동산 마케팅 콘텐츠 자동 생성 & 게시
   └─ 핵심 가치: 수작업 3~4시간/건 → 자동 15분/건

2. 전략 및 목표
   └─ North Star Metric: 월간 신규 적격 관심 유저 수
   └─ Input Metrics: 주 3회 발행, 게시물당 도달 500, 프로필 방문 5%, 행동 전환 20%

3. 에이전트 팀 구성
   ├─ Max (매니저): 일일 계획, KPI 추적
   ├─ Aria (콘텐츠): 카드뉴스 생성, 캡션 작성
   └─ Luna (검수): 품질 검사, 법률 준수

4. 파이프라인 흐름
   └─ 데이터 수집 → 선별 → 디자인 → 검수 → 게시

5. 품질 기준
   └─ 금지어 목록, 면책 문구, 이미지 사양
```

### [개념] 하네스 엔지니어링이 이 PRD에 어떻게 적용되었는지

2.2에서 배운 하네스 엔지니어링의 핵심 개념들이 이 PRD 안에 그대로 녹아 있어요.

| 하네스 개념 | PRD에서의 적용 |
|------------|---------------|
| **컨텍스트 불안 해결 (교대 근무)** | 하나의 AI가 모든 것을 하지 않고, Max → Aria → Luna 순서로 역할을 분리하여 각자의 영역에 집중합니다 |
| **자기 평가 편향 해결 (Generator/Evaluator 분리)** | Aria가 콘텐츠를 만들고(Generator), Luna가 별도로 검수합니다(Evaluator). 만든 AI와 평가하는 AI가 완전히 분리되어 있습니다 |
| **CLAUDE.md (업무 매뉴얼)** | PRD 자체가 에이전트들의 업무 매뉴얼 역할을 합니다. 각 에이전트가 무엇을 해야 하고, 어떤 기준을 지켜야 하는지 명시되어 있습니다 |
| **3-에이전트 패턴** | Max = Planner(기획), Aria = Generator(실행), Luna = Evaluator(검수)로 정확히 대응합니다 |

### [개념] 에이전트 정의서

PRD가 전체 시스템의 설계도라면, 에이전트 정의서는 **각 에이전트의 개인별 업무 매뉴얼**입니다. 각 에이전트가 무엇을 할 수 있고, 어떤 규칙을 따르는지 구체적으로 정의해요.

**예시: Aria (콘텐츠 에이전트)**

| 항목 | 내용 |
|------|------|
| **이름** | Aria (아리아) |
| **역할** | 콘텐츠 생성 전문가 |
| **담당 업무** | 카드뉴스 제작, 캡션 생성 |
| **사용 도구** | 데이터 조회 API, HTML 렌더러 |
| **입력** | 실거래가 데이터 (CSV) |
| **출력** | 카드뉴스 이미지 + 인스타 캡션 |
| **품질 기준** | 금지어 미포함, 면책 문구 포함, 1080x1080 |

**예시: Luna (검수 에이전트)**

| 항목 | 내용 |
|------|------|
| **이름** | Luna (루나) |
| **역할** | 품질 관리 전문가 |
| **담당 업무** | 규칙 검사, LLM 품질 평가, 승인/반려 |
| **판단 기준** | 80점 이상: 자동 승인 / 50~79점: 사람 검토 / 50점 미만: 반려 |

> 이 PRD와 에이전트 정의서가 2.4에서 직접 만들어볼 때의 참고 자료가 돼요.
> 걱정하지 마세요! 클로드 코드에게 목적과 기능을 간단하게 설명하면, PRD도 함께 AI-Native 방식으로 작성할 수 있어요. 처음부터 완벽하게 쓸 필요 없이, 대화하면서 함께 만들어가면 돼요.

| 옵션 | 설명 |
|------|------|
| **A. 디자인 작업 개선** | 기존에 수작업으로 하던 디자인 작업을 클로드로 개선 |
| **B. 데이터 + 디자인 파이프라인** | 데이터를 가져와서 자동으로 디자인 결과물 생성 |
| **C. 콘텐츠 호스팅 자동화** | 수동으로 올리던 콘텐츠를 자동 게시 |
| **D. 나만의 아이디어** | 다른 자동화하고 싶은 업무! |

> **D를 선택한 경우:** 클로드 코드에게 아래와 같이 입력하여 업무를 정리할 수 있습니다.
> ```
> "나는 [하고 싶은 업무]를 자동화하고 싶은데, 어떻게 시작하면 좋을지 모르겠어.
> 내 업무에 대해 질문해줘. 대화하면서 함께 정리하자."
> ```
> 클로드 코드가 질문을 던져주면, 대화하듯 답변하면서 업무를 구체화할 수 있어요.

---

### Step 2: 클로드와 함께 PRD 작성하기 (15분)

자동화할 작업이 정해졌으면, 아래 프롬프트를 클로드 코드에 입력해 주세요. 대괄호 `[ ]` 안의 내용은 본인의 상황에 맞게 바꿔서 입력해 주세요.

**[클로드 코드에 입력]**
```
"나는 [선택한 작업]을 자동화하고 싶어.

현재 상황:
- [현재 어떻게 작업하고 있는지]
- [어떤 부분이 반복적이고 시간이 오래 걸리는지]
- [최종 결과물은 어떤 형태인지]

이 작업을 자동화하기 위한 PRD를 작성해줘.
PRD에는 다음을 포함해줘:
1. 프로젝트 개요와 목적
2. 필요한 에이전트 역할 정의 (각각 이름, 역할, 담당 업무)
3. 작업 흐름 (파이프라인)
4. 입력 데이터와 최종 출력물
5. 품질 기준"
```

> 클로드가 PRD를 작성해주면, 내용을 읽어보고 수정이 필요한 부분을 대화로 요청하면 돼요.

---

### Step 3: 에이전트 구축하기 (20분)

PRD가 완성되면, 아래 프롬프트를 입력하여 실제 에이전트를 만들어달라고 요청해 보세요.

**[클로드 코드에 입력]**
```
"이 PRD를 바탕으로 에이전트를 구축해줘.

구축 순서:
1. 먼저 CLAUDE.md를 만들어줘 (이 프로젝트의 규칙과 역할 정의)
2. 가장 핵심이 되는 파이프라인부터 만들어줘
3. 한 단계씩 테스트하면서 진행하자"
```

---

### Step 4: 테스트 & 반복 개선 (남은 시간)

에이전트가 만들어지면, 실제로 작동하는지 테스트해 보세요.

**[클로드 코드에 입력]**
```
"방금 만든 에이전트를 테스트해보자.
[테스트 데이터 또는 상황]을 입력으로 실행해줘."
```

결과물을 확인한 후, 개선이 필요한 부분이 있다면 아래와 같이 요청해 보세요.

**[클로드 코드에 입력]**
```
"결과물을 보니 [개선할 점]이 있어.
[구체적으로 어떻게 바꿔야 하는지] 수정해줘."
```

---

### 반복 개선 시 스스로 점검해볼 질문들

아래 질문들은 클로드에게 입력하는 것이 아니라, **결과물을 보면서 스스로 점검하는 질문**이에요.

- "결과물의 품질이 내 기대에 맞는가?"
- "아직 수작업인 단계가 남아 있는가?"
- "다른 사람이 이 결과물을 바로 쓸 수 있는가?"

점검 결과 개선이 필요하면, 위의 테스트 & 개선 프롬프트를 반복해 주세요.
